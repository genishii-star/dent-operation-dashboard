# Airbnb レビュー自動化 / 分析システム 仕様書 v0.2

作成日: 2026-04-07
作成者: 石井 + Claude
ステータス: ドラフト（オペレーションチーム共有用）

---

## 1. 目的

1. Airbnb の **ホスト→ゲスト レビュー** を自動投稿し、運営工数をゼロにする
2. 受信レビュー（ゲスト→ホスト）・プライベートフィードバック・★スコアを一元集約
3. ★低評価レビューへの返信をドラフト生成し、承認フローで投稿
4. すべてを既存 operation dashboard（`index.html` のレビュータブ）に統合

## 2. システム全体像

本システムは **2つの独立した系統** で構成される。

```
┌─────────────────────────────────────────────────┐
│  系統A: 投稿系（毎日）                          │
│                                                  │
│  既存Chrome拡張「Dent データ同期」に統合        │
│  ワンクリック同期ボタン押下時に併走             │
│                                                  │
│  - Airhost「評論を書く」UIから自動投稿          │
│  - 1セッションで全Airbnbアカウント対応          │
│  - Airhost側で催促OFFを自動判定                 │
└─────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────┐
│  系統B: 取得系（週1）                           │
│                                                  │
│  Playwright + ローカルMac + launchd              │
│                                                  │
│  - Airbnb直接スクレイピング                     │
│  - 受信レビュー / ★ / プライベートフィードバック│
│  - 低評価返信のドラフト生成→Slack承認           │
└─────────────────────────────────────────────────┘
```

両系統は独立してデプロイ・運用可能。系統Aだけで「自動投稿」のメイン要件は満たせる。

## 3. 系統A: 投稿系（Chrome拡張統合）

### 3.1 トリガー
既存の Chrome 拡張ポップアップ「ワンクリック同期」ボタン押下時に併走。
ユーザー側の運用は**現状から1ステップも増えない**。

### 3.2 フロー

```
[ワンクリック同期ボタン押下]
  ↓
1. 既存: 日次データCSV取得 → スプシ書き込み
2. 既存: 予約データCSV取得 → スプシ書き込み
3. 【新規】チェックイン管理ページを開く
4. 【新規】未投稿レビューの予約を抽出
5. 【新規】各予約について:
   a. 催促OFFチェック（Airhost UI上で判定）
   b. ゲスト言語判定（名前・国籍・予約言語から）
   c. Claude API で本文生成
   d. 「評論を書く」モーダルを開いて入力→送信
      - ★3軸（清潔さ/チェックイン心得/コミュニケーション）すべて5固定
      - 公開コメント（1000字以内、自動生成）
      - 推薦するか: Yes 固定
      - プライベートフィードバック: 空欄（Phase 1では未対応）
   e. 結果をスプシ「レビュー実行ログ」に追記
6. 既存: Slack通知（同期+レビュー件数のサマリー）
```

### 3.3 レビュー本文生成仕様

| 項目 | 仕様 |
|---|---|
| 生成元 | Claude API（Sonnet推奨、コスト最適化） |
| 言語 | ゲスト名・国籍・予約言語から自動判定（日/英/中、デフォルト英） |
| 文字数 | 各言語で2〜4文程度 |
| 内容 | 滞在日数・人数・物件名を反映、ブランドトーンに従う |
| ★評価 | 全軸5固定 |
| 推薦 | Yes 固定 |

### 3.4 実装範囲

**変更ファイル:**
- `chrome-extension/inject.js`: チェックイン管理ページの操作・モーダル制御
- `chrome-extension/content.js`: 既存同期フローへの組み込み
- `chrome-extension/popup.html` / `popup.js`: 進捗表示の追加
- `chrome-extension/manifest.json`: Claude API用の host_permissions 追加

**新規:**
- `chrome-extension/review-poster.js`: レビュー投稿ロジック
- `chrome-extension/review-generator.js`: Claude API呼び出し・言語判定

### 3.5 安全機構

| 機構 | 内容 |
|---|---|
| 様子見モード | 初期は「ドラフト生成のみ・実投稿しない」フラグで運用 |
| ホワイトリスト | 特定物件のみ自動投稿対象 → 段階的に拡大 |
| 投稿件数上限 | 1回の同期で最大N件（N=10程度） |
| 失敗時の継続 | 1件失敗しても次の予約へ進む |
| 重複防止 | 投稿済み予約IDをスプシで管理、再投稿しない |

| 項目 | 仕様 |

| 実行方法 | launchd（毎週月曜 AM 3:00） |

| 実行ディレクトリ | `/Users/genishii/Client/claude-code/operation/review/` |

### 4.2 機能一覧

#### A. データ取得

| # | 機能 | 取得元 |
|---|---|---|
| A1 | 受信レビュー本文取得 | `/hosting/reviews` |
| A2 | ★スコア取得（カテゴリ別6軸） | 同上 |
| A3 | プライベートフィードバック取得 | レビュー詳細 |
| A4 | ゲスト名・予約ID・物件・日付 | 同上 |
| A5 | レビュー記載率（投稿された/されてない比率） | 物件統計ページ |
| A6 | 全体評価サマリー（リスティング単位） | リスティング管理ページ |

#### B. データ書き込み

| # | 機能 | 備考 |
|---|---|---|
| B1 | レビュー返信の自動投稿 | **★4以上は完全自動 / ★3以下はSlack承認後に投稿** |

#### C. 分析（Claude APIで処理）

| # | 機能 | 処理 |
|---|---|---|
| C1 | ネガポジ判定 | 全件をpos/neu/negに分類 |
| C2 | 頻出キーワード抽出 | ポジ・ネガそれぞれの上位語 |
| C3 | プライベートフィードバック要約 | 物件別に「最近の指摘事項TOP3」を生成 |
| C4 | 物件別トレンド計算 | 月次★推移、レビュー数推移 |
| C5 | アラート判定 | ★急落・ネガ急増を検知してSlack通知 |

### 4.3 実行フロー

```
[毎週月曜 AM 3:00 launchd 起動]
  ↓
for each account in sessions/*.json:
  1. セッション復元
  2. Airbnb /hosting/reviews を開く
  3. 受信レビュー全件取得（A1-A4）
  4. プライベートフィードバック取得（A3）
  5. リスティング統計取得（A5, A6）
  6. スプシ書き込み
  ↓
全件まとめて分析:
  7. Claude APIでネガポジ・キーワード分析（C1, C2）
  8. プライベートフィードバック要約（C3）
  9. トレンド計算（C4）
  10. アラート判定（C5）
  ↓
返信処理:
  11. ★4以上で未返信 → Claude生成 → 自動投稿（B1）
  12. ★3以下で未返信 → Claude生成 → Slack承認カード送信
  13. 前回承認済みの返信 → 投稿
  ↓
完了通知:
  14. Slack に実行サマリー送信
```

### 4.4 セッション管理

**初回セットアップ**（1アカ約1分、計10分）:
```
npm run login -- --account=account-01
→ Playwrightがブラウザを起動
→ 手動でログイン（2FAも手動）
→ sessions/account-01.json に保存
```

**通常運用:**
- セッションは数ヶ月持つはず（要検証）
- 切れたらSlackに `account-NN のログインが切れました` と通知
- 該当アカだけ再ログインコマンドを実行

### 4.5 ★低評価返信の承認フロー

```
[週1取得系実行]
  ↓
★3以下のレビュー検出
  ↓
Claudeで返信文ドラフト生成
  ↓
Slackに承認カード送信（Block Kit）
  ┌────────────────────────────────┐
  │ 物件: ENK #801                  │
  │ ゲスト: John Smith              │
  │ ★: 2/5                          │
  │ 本文: "...最悪だった..."        │
  │                                 │
  │ 提案返信:                        │
  │ "このたびはご不便を..."          │
  │                                 │
  │ [承認] [編集] [却下]            │
  └────────────────────────────────┘
  ↓
承認 → 翌週の取得系実行時にAirbnbに投稿
編集 → Dashboardに飛ばして編集→保存→承認待ちに戻す
却下 → 返信なし
```

- 承認状態はスプシで管理
- Dashboardの「承認待ち」一覧でも同じものが見える

## 7. セキュリティ要件

- Airbnbのセッション情報（`sessions/*.json`）は **git にコミット禁止**
- `.gitignore` に `sessions/` を追加
- macOS Keychain or sops で暗号化保存
- Claude APIキーは環境変数管理
- Slack Webhook URL も環境変数管理（過去にコミット履歴あり、再生成要）

## 8. リスクと対策

| リスク | 影響 | 対策 |
|---|---|---|
| Airbnb の利用規約違反判定 | アカウントBAN | 系統Aは Airhost UI 経由でリスク低、系統Bは週1の低頻度 |
| Airhost の DOM 変更 | 投稿失敗 | セレクタ設定ファイル化・Slack通知 |
| Airbnb の DOM 変更 | 取得失敗 | 同上 |
| 2FA による自動化失敗 | ログイン不可 | 初回手動ログイン → セッション長期保持 |
| LLM 生成文の品質ばらつき | ゲストへの不適切メッセージ | 様子見期間で完全手動承認 |
| 低評価返信の誤承認 | 火に油 | Claude生成 + 人間承認の二重化 |

## 12. 確定事項（v0.2 時点）

- 投稿系（系統A）= 既存Chrome拡張のワンクリック同期ボタンに統合
- 取得系（系統B）= Playwright + Mac + 永続セッション + 週1 launchd
- ★評価は全軸5固定
- 言語は自動判定（日/英/中）
- ★4以上の返信は完全自動投稿
- ★3以下の返信は Slack承認 + Dashboard承認の二系統
- プライベートフィードバックは取得対象に含む
- Chromeプロファイル分けへの移行は**しない**（オペチーム慣れ優先）
